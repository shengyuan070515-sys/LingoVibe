# 每日阅读（替换每日播客）— 产品设计规格

**状态**：已评审锁定（2026-04-04）  
**范围**：第一版（v1）行为与架构基线；实现计划见后续 `docs/superpowers/plans/`。

---

## 1. 背景与目标

将现有「每日播客」整体替换为「每日阅读」：用户从**文章库**选文阅读，支持**联网检索入库**与**本地导入**；阅读器提供翻译、语法、朗读、生词本等能力；**多档完成度**与**每日闭环**挂钩规则如下文锁定。

**非目标（v1 不做）**

- 播客相关页面与入口的任何保留或过渡。
- 用户通过链接自动抓全文（导入仅粘贴 / `.txt`）。
- PDF / EPUB 导入。
- 测验功能的完整可用（仅占位 UI）。

---

## 2. 信息架构与核心流程

### 2.1 入口与列表

- 用户进入「每日阅读」后看到**多篇文章**可选（非单篇直达）。
- 文章来源两类：
  1. **联网精选**：经检索 → 候选列表 → 用户勾选 → 入库。
  2. **用户导入**：粘贴纯文本或上传 `.txt`。

### 2.2 联网检索

- **真实联网检索**；结果需能进入本地/持久化**文章库**（实现形态由工程选型，见第 7 节）。
- 发起方式：**关键词 / 一句话主题**为主；**快捷主题标签**为辅（本质是预填检索词）。
- **不以「纯一键今日推荐」作为唯一入口**（v1）。

### 2.3 候选入库交互

- 检索返回多条候选时：用户**手动勾选一篇或多篇**后再入库。
- **出处硬门槛**：须具备可展示的 **规范 URL** 与 **标题类信息**；不满足则**不进入正式文章库**（可另设失败/待修复队列，但不混入正式列表）。

### 2.4 去重

- 以**规范化后的 canonical URL** 为键；**禁止**同一 URL 多条并存。
- 重复选择时：**进入已有篇目**；产品保留「用本次抓取**刷新正文快照**」的能力空间（实现可分步：先提示 + 进入，刷新为紧随迭代）。

### 2.5 难度

- **系统/AI 在入库时给默认难度**；**用户可手动覆盖**。
- 用户覆盖后：**界面不再展示**系统原始难度；对外仅展示当前有效难度。
- **第一版统一 5 档**：入门 / 基础 / 中级 / 进阶 / 高阶（存储可用 1–5 或枚举）。

### 2.6 旧「每日播客」

- **不保留**旧入口或过渡页；产品层面**直接替换**为「每日阅读」。
- 升级后：**自动清理**与播客相关的本地持久化数据（具体 key 由实现枚举并写迁移说明）。

## 3. 完成度分档与每日闭环

### 3.1 分档理念

- **多档完成度**：不同指标对应不同档位；用户自主追求更高档，产品为每档设**可自动判定**门槛。
- **无全站兜底**：不提供「一键今日已读」类按钮来完成闭环。

### 3.2 最低档 — 浏览完成（驱动每日闭环）

- **每日闭环**：当日在**任意一篇**文章上达成**最低档**，即视为完成「每日阅读」相关闭环（不绑定某篇「今日指定文」）。
- **判定 = 全文覆盖/到达文末** + **仅统计前台、标签页可见的停留时间**。
- **最短停留秒数**（\(W\) 为正文英文词数；中文正文采用「等效词数」估算，常数在实现阶段确定并写入代码注释）：

\[
D = \mathrm{clamp}(\mathrm{round}(0.06 \cdot W),\ 40,\ 180)
\]

含义：按 250 wpm 粗估读完时间，取其约 **25%** 作为浏览档停留下限；**下限 40s、上限 180s**。

- 存在分页或「展开全文」时，以**最终展开后的全文**为测量与滚动目标。

### 3.3 更高档 — 测验完成（v1 占位，规则先行）

- v1 **测验 UI 占位**，但规格预先锁定以便后续实现：
  - **最少 5 题**方可进入可提交/可判定状态；不足则测验档不可用或不可判完成。
  - **通过条件**：`答对题数 × 5 ≥ 总题数 × 4`（等价正确率 ≥ 80%）。
  - 用户可**查看解析**、**重做**；这些行为**不参与**完成判定（判定仅看是否满足上述通过条件）。
- 测验启用后的「通过后是否允许重测覆盖状态」等细项，在测验功能落地时单独补一小节规格（默认倾向：通过后保持已通过，重做仅用于学习）。

## 4. 文章详情页（阅读器）— v1 能力

### 4.1 必须有

| 能力 | 行为摘要 |
|------|----------|
| 翻译 | **整篇**与**选区**两种入口；**默认仅展示原文**；译文需用户**点击按钮**后展开；默认主路径侧重**选区翻译**，整篇为另一明确按钮。 |
| 语法分析 | **仅对选中**文本进行分析（不做整篇默认一次性分析）。 |
| 朗读/播放 | **整篇**与**选区**均支持，以**不同按钮**区分。 |
| 生词本 | 与现有生词本能力打通（具体交互与主站一致优先）。 |

### 4.2 占位

- **测验**：v1 灰显或「即将推出」，不实现完整判分链路。

---

## 5. 设置与密钥

- **每日阅读模型能力**（翻译、语法等调用 LLM 的部分）：使用 **独立 Settings 字段**（与「AI 对话」Key **分离**）。
- **联网搜索**：用户在 **Settings** 配置 **Search API Key**；每次检索由前端将 Key 传给 **Serverless**（或等价安全传输方式），**服务端不落库、不写入持久化日志**（实现需避免在日志中打印明文 Key）。

---

## 6. 联网架构基线（检索 + 正文）

- **不以纯浏览器**承担「关键词搜全网 + 稳定入库」主路径（搜索 Key 与 CORS 限制）。
- **Serverless 边缘函数（或同类）** 作为 **检索与编排层**。
- 函数内部 **优先使用 Jina Reader 或 Firecrawl 等** 将 URL 转为干净正文/Markdown；抽取失败时的降级策略在实现阶段定义（可选：第二抽取器、错误提示、不入库）。

**工程待定（不阻塞规格一致性）**

- 具体 Search API 供应商、Serverless 托管平台、Jina 公共端点策略变更时的 fallback。

---

## 7. 数据与信任（产品层）

- 入库文章建议字段（实现可增减）：`canonicalUrl`、`sourceTitle`、`fetchedAt`、`contentSnapshot`（或分段存储）、`difficulty`（有效值）、`sourceType`（`web_curated` | `user_import`）。
- 用户导入：**允许无 URL**，`sourceType = user_import`。
- 阅读页需有简短 **信任/数据说明**（模型 Key、搜索 Key 经 Serverless、语音若使用浏览器 API 等——文案与实现一致即可）。

---

## 8. 自检（规格一致性）

- [x] 闭环仅依赖最低档；与「无全站兜底」一致。
- [x] 联网文必须出处；用户导入可无 URL 但有类型标记。
- [x] 去重与手动多选入库不冲突。
- [x] 测验规则与 v1 测验占位不矛盾（测验未上线时不产生第二档完成记录）。
- [x] 旧播客移除与本地数据清理已写明。
