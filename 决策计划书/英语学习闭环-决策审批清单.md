# 英语学习产品闭环 — 按顺序审批的决策清单

下面是一份**按顺序审批的决策清单**。每一条都写明「在定什么」「可选方向」「逻辑约束」「可行性说明」，你可以从上到下逐条拍板；前一条会约束后一条，避免方向打架。

---

## A. 战略与边界（先定盘子）

**A1 — 第一期「闭环」是否只认一条主轴？**  
- **含义**：资源集中做透一条路径，其它功能不扩或只做入口。  
- **选项**：  
  - **A1a** 是，主轴 = 生词本（收录 → 复习 → 再遇见）  
  - **A1b** 是，主轴 = AI 对话练口语  
  - **A1c** 否，两条主轴并行投入  
- **逻辑**：选 A1c 会拉长周期、易做成「都有、都不深」。  
- **可行性**：A1a 与现有数据结构（生词、SRS 字段）最贴合；A1b 依赖对话质量与反馈设计，单独做闭环更难。  
- **建议倾向**：A1a（除非你明确想主打口语品牌）。

**A2 — 「第一期」时间盒多长？**  
- **选项**：例如 2 周 / 4 周 / 8 周（你填数字即可）。  
- **逻辑**：时间盒决定 A3 能塞进多少项。  
- **可行性**：阶段 1（今日任务 + 复习入口 + 一种复习形态）在 2～4 周内可做到可用；含造句批改更偏 4～8 周。

**A3 — 第一期明确「不做」什么？（写进纪律）**  
- **示例选项**（可多选「本期不做」）：假进度课程、重社区、多复习题型并行、新用户体系/登录、复杂成就体系……  
- **逻辑**：防止闭环被稀释；和 A1、A2 一致。  
- **可行性**：列出 3～5 条「不做」比堆功能更省返工。

---

## B. 用户与场景（谁在什么情境下用）

**B1 — 核心用户画像（第一期只服务一种）**  
- **选项**：  
  - **B1a** 碎片化自学（地铁、睡前）  
  - **B1b** 备考导向（雅思/托福等）  
  - **B1c** 职场商务  
- **逻辑**：影响文案、题型、难度默认、播客/对话语气。  
- **可行性**：都可做，但**不要第一期三种都要**。

**B2 — 单次日均投入预期**  
- **选项**：例如 5 分钟 / 15 分钟 / 30 分钟。  
- **逻辑**：直接决定「今日任务」几步、每步体量。  
- **可行性**：5～15 分钟与「复习 5 词 + 轻互动」最匹配。

---

## C. 闭环定义（产品上什么叫「完成一圈」）

**C1 — 「今日闭环完成」的必备条件包含哪几项？**（勾选制，至少 1 项，建议 2～3 项）  
- **C1a** 完成当日待复习队列（或完成固定 N 个）  
- **C1b** 完成至少 1 轮 AI 对话（或最少条数）  
- **C1c** 完成 1 次播客/听读（或最小时长）  
- **C1d** 收录 ≥1 个新词（容易变强迫，慎选）  
- **逻辑**：项越多，完成率越低；**复习**应优先于「再输入」。  
- **可行性**：C1a 最可控；C1b/c 依赖用户是否当天打开对应模块。

**C2 — 「待复习」的词从哪里来？**  
- **选项**：  
  - **C2a** 仅用现有 SRS：`nextReviewDate ≤ 今天`  
  - **C2b** 再加「手动标为重点」插队  
  - **C2c** 再加「错题本」队列（需定义何为错题）  
- **逻辑**：C2a 实现成本最低且和现有字段一致；C2b/c 增量清晰。  
- **可行性**：先做 C2a，通过后两条再批。

---

## D. 复习形态（闭环的核心能力）

**D1 — 第一期唯一复习形态选哪种？**  
- **选项**：  
  - **D1a** 闪卡（英↔中，自评会/不会或三档）  
  - **D1b** 四选一选择题（释义/搭配）  
  - **D1c** 拼写/听写  
- **逻辑**：D1a 开发量小、适合碎片；D1c 最重。  
- **可行性**：D1a + 更新 `interval` / `nextReviewDate` / `level` 最顺。

**D2 — 自评结果如何映射到 SRS？**  
- **选项**：  
  - **D2a** 简单规则表（会→间隔×2，不会→重置/缩短）  
  - **D2b** 先固定间隔（如 1/3/7 天阶梯），不做复杂 SM-2  
- **逻辑**：规则要先定死，再谈算法优化。  
- **可行性**：D2b 最易解释、最易调试。

**D3 — 复习是否只做「单词」，还是「例句」也进队列？**  
- **选项**：**D3a** 仅 `type === 'word'`；**D3b** 例句也参与（需单独规则）。  
- **逻辑**：和生词本数据模型一致。  
- **可行性**：D3a 风险低。

---

## E. 输出与反馈（从「认」到「用」放第几期）

**E1 — 第一期要不要强制「输出任务」？**  
- **选项**：**E1a** 不要，复习只做识别；**E1b** 要，每词或每轮一条最小输出。  
- **逻辑**：E1b 显著增加产品与模型/批改成本。  
- **可行性**：E1a 更易按期交付；E1b 适合第二期。

**E2 — 若上输出，优先形态？**（仅当 E1b 选「要」时才审）  
- **选项**：造句 + 文本反馈 / 跟读 + 自评 / 选择题。  
- **可行性**：选择题 < 跟读自评 < 造句+AI 批改（成本递增）。

---

## F. 与 AI / 播客的联动（增强闭环，非第一期必做）

**F1 — AI 对话是否在系统层注入「今日待复习词」？**  
- **选项**：**F1a** 否；**F1b** 是（开场或提示里自然出现 1～2 词）。  
- **逻辑**：F1b 提高「再遇见」概率，但要控提示词长度与重复烦感。  
- **可行性**：F1b 多为 prompt/上下文拼接，工程中等。

**F2 — 播客是否写入「今日闭环」？**  
- **选项**：**F2a** 写入；**F2b** 不写入，仅作选修。  
- **逻辑**：与 C1 一致；播客完成判定（听完？生成？）要产品定义。  
- **可行性**：F2b 更简单；F2a 需明确完成条件。

---

## G. 入口与信息架构（用户找得到）

**G1 — 「开始复习」主入口放哪？**  
- **选项**：首页模块 / 生词本顶部固定按钮 / 底栏第 2 项改为复习（会动现有导航）/ 多入口一致状态。  
- **逻辑**：**只能有一个「主」入口**，其它为快捷方式。  
- **可行性**：首页卡片 + 生词本顶部最直观。

**G2 — 首页「今日任务」与现有「活跃度/连胜」关系？**  
- **选项**：**G2a** 并行展示、完成今日任务可加活跃度；**G2b** 逐步以今日任务为主文案。  
- **逻辑**：避免两个体系各说各话。  
- **可行性**：G2a 兼容现状，改动小。

---

## H. 数据、隐私与降级（保证能上线）

**H1 — 无 API Key / 超限时，复习是否仍可离线完成？**  
- **选项**：**H1a** 是，复习不依赖大模型；**H1b** 否。  
- **逻辑**：闭环不应被 Key 卡死。  
- **可行性**：H1a 应作为硬要求。

**H2 — 是否记录「复习日志」供以后分析？（本地即可）**  
- **选项**：要 / 不要 / 第二期再说。  
- **逻辑**：要日志利于以后调 SRS，但增加一点存储与迁移考虑。  
- **可行性**：本地 persist 可行。

## I. 审批顺序建议（你怎么用这份清单）

1. 先批 **A1 → A2 → A3**（主轴、时间、不做清单）  
2. 再批 **B1 → B2**（用户与日投入）  
3. 再批 **C1 → C2**（什么叫今日完成、队列规则）  
4. 再批 **D1 → D2 → D3**（复习形态与 SRS）  
5. 然后 **E1（及必要时 E2）**  
6. 再 **F1 → F2**  
7. 最后 **G1 → G2 → H1 → H2**
